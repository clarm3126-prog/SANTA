# SANTA Research

## 1. 한 줄 정의
이 저장소는 빌드 도구 없이 실행되는 정적 웹 프로토타입이며, 현재는 `등산 정보 + 커뮤니티 + 배지 + 선택적 동행` 구조를 검증하는 단계다.

핵심 변화는 이미 일어났다. 처음 구조는 동행 중심 시연에 가까웠지만, 현재 코드는 홈 정보 구조와 일부 데이터 읽기 경로를 커뮤니티 중심으로 재배치했다.

## 2. 저장소 구조
현재 루트 기준 핵심 파일은 아래와 같다.

- `index.html`: 단일 HTML 엔트리 포인트
- `assets/css/styles.css`: 전체 스타일시트
- `assets/js/app.js`: 앱 데이터, 렌더링, 이벤트, Supabase 읽기 로직
- `assets/js/supabase-config.js`: Supabase URL/anon key 설정 파일
- `supabase/migrations/202604020001_phase1_core.sql`: 1단계 코어 스키마
- `supabase/migrations/202604020002_seed_mountains_routes.sql`: 산/코스 시드
- `docs/current-status.md`: 현재 작업 체크포인트
- `docs/phase-1-supabase-guide.md`: 초보자용 실행 가이드
- `plan.md`: 현재 기준 개발 계획

## 3. 현재 앱이 실제로 어떻게 동작하는가

## 3.1 부팅 흐름
브라우저에서 `index.html`을 열면 다음 순서로 실행된다.

1. 폰트와 스타일을 먼저 읽는다.
2. 화면 골격과 각 `screen` 섹션이 정적으로 렌더된다.
3. Supabase CDN을 읽는다.
4. `assets/js/supabase-config.js`를 읽는다.
5. `assets/js/app.js`가 실행된다.
6. `initApp()`이 즉시 실행된다.
7. 초기 렌더 후, Supabase 설정이 있으면 산/게시글 데이터를 비동기로 읽어와 다시 렌더한다.

즉 이 앱은 `정적 초기 렌더 -> 비동기 데이터 덮어쓰기` 방식으로 움직인다.

## 3.2 화면 구조
`index.html`은 크게 두 영역이다.

- `aside.overview`: 데스크톱용 설명 패널
- `section.phone-stage > #app.shell`: 실제 앱처럼 보이는 인터랙티브 영역

앱 내부 화면은 아래 다섯 개다.

- `screen-home`
- `screen-companion`
- `screen-detail`
- `screen-create`
- `screen-profile`

라우터는 없고, `.screen.active` 클래스를 토글해서 전환한다. 즉 URL 상태 공유나 브라우저 히스토리 기반 탐색은 없다.

## 3.3 자바스크립트 구조
`assets/js/app.js`는 현재 크게 여섯 층으로 읽는 것이 맞다.

1. `APP_DATA`: 시연용 샘플 데이터
2. `runtimeData`: 샘플 데이터를 복제한 런타임 데이터
3. Supabase 설정과 클라이언트 생성
4. `state`: 현재 선택된 산과 검색어 같은 최소 화면 상태
5. 매퍼/조회/렌더 함수
6. 이벤트 위임과 초기화

이 구조의 핵심은 `APP_DATA`를 완전히 버리지 않았다는 점이다. 지금 코드는 서버가 없는 데모와 실제 DB 읽기 모드를 둘 다 지원하려는 과도기 상태다.

## 3.4 데이터 흐름
현재 읽기 경로는 아래처럼 나뉜다.

### 샘플 데이터 전용
- `companions`
- `profile`
- 일부 상세용 시각 데이터 fallback

### DB 읽기 가능
- `mountains`
- `routes`
- `posts`
- `profiles` 일부 조인 정보
- `post_images`

즉 실제 앱 상태는 `전부 DB 기반`이 아니라 `산과 게시글은 DB 우선, 나머지는 더미 유지` 상태다.

## 4. HTML 관점 분석

## 4.1 홈
홈은 현재 이 앱의 가장 중요한 화면이다. 구조는 아래 순서로 읽힌다.

- 상단 브랜딩과 검색
- 산 카드 목록
- 커뮤니티 브리핑
- 지도 브리핑
- 선택적 동행 블록

이 순서는 서비스 방향과 맞다. 사용자가 먼저 `어디 갈지`, `사람들이 뭐라고 하는지`, `코스가 어떤지`를 보고, 그 다음 필요하면 동행으로 내려가게 설계되어 있다.

## 4.2 커뮤니티 화면
`screen-companion`은 이름은 예전 구조의 흔적이지만, 현재 역할은 커뮤니티 허브에 가깝다. 다만 파일 이름과 DOM id는 아직 예전 명명법을 유지해서, 문서와 실제 의미 사이에 약간의 괴리가 있다.

## 4.3 상세 화면
상세 화면은 산 하나를 중심으로 `개요`, `스탯`, `지도/코스`, `커뮤니티`, `생활 정보`를 붙인 구조다. 이 화면은 장기적으로 SANTA의 핵심 화면이 될 가능성이 높다.

## 4.4 작성 화면
작성 화면은 구조상 존재하지만, 아직 실서비스 기능으로는 반쯤만 연결되어 있다. 폼을 제출해도 DB insert가 아니라 토스트만 뜨고 리셋된다.

## 4.5 프로필 화면
프로필 화면은 시각적으로는 거의 완성형이지만, 실제 데이터 관점에서는 가장 더미 의존도가 큰 화면이다. 아직 `profiles`, `user_badges` 연결이 안 되어 있다.

## 5. CSS 관점 분석

## 5.1 스타일 시스템 성격
스타일은 단일 글로벌 CSS 파일에 모두 들어 있다. 컴포넌트 단위 분리는 없지만, 토큰과 카드 시스템은 어느 정도 통일돼 있다.

주요 특징은 아래와 같다.

- 숲/흙/안전 톤의 색상 시스템
- 앱 셸처럼 보이는 라운드 프레임
- 설명 패널과 앱 패널을 나란히 두는 데스크톱 시연 구조
- 모바일과 데스크톱을 모두 고려한 2단 브레이크포인트 전략

## 5.2 데스크톱 조정 상태
이전에는 폰 프레임 중심 폭이 강했지만, 현재는 데스크톱에서 더 넓게 보이도록 조정돼 있다.

핵심 변화는 아래다.

- 페이지 최대 폭 확장
- 앱 셸 최대 폭 확장
- 산 카드와 지도 브리핑의 다열 레이아웃
- 작성 폼과 본문 패딩 확대

즉 지금은 `휴대폰 목업만 보이는 데모`가 아니라, 컴퓨터에서도 어느 정도 웹처럼 읽히는 상태다.

## 5.3 남아 있는 CSS 부채
- 스타일 파일이 여전히 크고 단일 파일이다.
- 예전 동행 중심 구조에서 남은 클래스 명이 일부 있다.
- 컴포넌트 단위 재사용보다는 화면 단위 누적 스타일이 많다.

하지만 지금 단계에서는 구조 재작성보다 기능 연결이 우선이다.

## 6. 자바스크립트 관점 분석

## 6.1 데이터 모델의 성격
`APP_DATA` 안의 객체들은 서버 정규화 모델이 아니라, `카드 렌더링에 바로 쓰기 좋은 화면 모델`에 가깝다. 예를 들어 산 데이터 하나에 카드 메타, 상세 통계, 생활 정보, 시각 클래스가 함께 들어 있다.

이 방식은 시연에는 좋지만, 실제 DB와 1:1로 대응되지는 않는다. 그래서 현재 코드에는 DB 응답을 프런트 모델로 바꾸는 매퍼 함수가 들어가 있다.

## 6.2 중요한 함수 흐름
핵심 함수 흐름은 아래처럼 이해하면 된다.

- `loadMountainsFromSupabase()`: `mountains`와 `routes`를 읽고 홈/상세용 산 모델로 변환
- `loadPostsFromSupabase()`: `posts`, `profiles`, `mountains`, `routes`, `post_images`를 읽고 커뮤니티 카드 모델로 변환
- `renderMountainCards()`: 홈 산 카드 렌더
- `renderFeedPosts()`: 홈/상세 커뮤니티 카드 렌더
- `renderDetailScreen()`: 현재 선택된 산의 상세 화면 렌더
- `renderProfile()`: 현재는 샘플 프로필 렌더
- `renderApp()`: 여러 화면을 한 번에 재렌더
- `initApp()`: 초기 렌더 후 비동기 데이터 로드 시작

즉 구조상 중심 함수는 `renderApp()`과 `initApp()` 두 개다.

## 6.3 현재 상태 관리
현재 `state`는 크지 않다. 대표적으로 아래만 관리한다.

- `activeMountainId`
- `mountainSearch`

이 말은 곧, 아직 앱이 복잡한 클라이언트 상태를 가지지 않는다는 뜻이다. 반대로 말하면, 작성 상태, 좋아요 상태, 저장 상태, 인증 사용자 상태 같은 실제 서비스 상태는 아직 본격적으로 들어오지 않았다.

## 6.4 이벤트 처리 방식
이 앱은 프레임워크 대신 이벤트 위임을 사용한다. 문서 클릭 이벤트 하나에서 `data-action`, `data-nav`, `data-filter` 등을 읽어서 화면 전환과 토글을 처리한다.

이 방식의 장점은 간단하다는 점이다. 단점은 기능이 늘수록 분기문이 커지고, 요소와 로직의 연결이 문자열 규약에 의존한다는 점이다.

## 6.5 현재 가장 중요한 기술적 특징
현재 코드는 `샘플 데이터 앱`에서 `실데이터 읽기 앱`으로 이동 중인 과도기 구조다. 완전한 리팩터링보다 점진적 치환을 선택했고, 이 선택은 지금 상황에서는 합리적이다.

이유는 아래와 같다.

- 기획과 UI를 동시에 유지할 수 있다.
- DB 연결 실패 시에도 시연이 가능하다.
- 산과 커뮤니티부터 붙이고 나머지는 뒤로 미룰 수 있다.

## 7. 현재 DB 설계와 코드의 연결 상태

## 7.1 잘 맞는 부분
현재 합의된 1단계 DB와 코드는 방향이 잘 맞는다.

- `mountains` + `routes`는 홈과 상세의 중심 데이터다.
- `posts` + `post_images`는 홈 브리핑과 상세 커뮤니티 탭의 중심 데이터다.
- `profiles`는 게시글 작성자 표시와 이후 프로필 화면으로 확장된다.
- `badges` + `user_badges`는 차별화 포인트로 자연스럽게 붙는다.

## 7.2 아직 연결이 안 된 부분
- `profiles` 실제 프로필 읽기
- `user_badges` 읽기와 노출
- `comments` 읽기/쓰기
- `post_likes` 토글
- `bookmarks` 저장
- `activities` 기반 기록 연결
- `meetups` 기반 동행 연결

## 7.3 설계상 주의점
프런트의 산 `id`와 DB의 산 `id`는 현재 동일하지 않을 수 있다. 그래서 프런트는 DB 응답을 읽을 때 이름 기반 fallback을 활용해 안정적인 화면 id를 만든다. 이 방식은 과도기에는 유용하지만, 장기적으로는 `DB id 기반 라우팅`으로 정리하는 편이 더 깔끔하다.

## 8. 강점
- 제품 방향이 코드 구조에 어느 정도 반영돼 있다.
- 홈 IA가 서비스 가치 제안과 맞는다.
- 정적 데모와 DB 연결 모드를 함께 유지하고 있다.
- 데스크톱 시연성이 개선됐다.
- DB 1단계 범위가 현재 제품 핵심과 잘 맞는다.

## 9. 약점과 위험
- 단일 `app.js`에 데이터, 매퍼, 렌더, 이벤트가 몰려 있다.
- `companions`, `profile`은 아직 더미라서 전체 일관성이 완전하지 않다.
- 작성/좋아요/저장 같은 핵심 성장 기능이 아직 읽기 전용에 가깝다.
- 화면 이름과 실제 역할이 일부 어긋난다. 예를 들어 `screen-companion`은 현재 커뮤니티 허브에 가깝다.
- `APP_DATA`와 DB 데이터가 함께 존재해 개념이 헷갈릴 수 있다.

## 10. 지금 가장 먼저 손대야 하는 지점
기술적으로 가장 먼저 손대야 할 지점은 아래 네 군데다.

1. 게시글 작성 폼을 `posts` insert와 연결
2. `post_likes`, `comments`, `bookmarks` 연결
3. `profiles`, `user_badges`로 프로필 화면 교체
4. `companions`를 메인 가치가 아닌 보조 모듈로 더 약하게 배치

## 11. 초보자용 이해 포인트
- 지금 앱은 완전한 웹앱이 아니라, 정적 프로토타입에 실DB 읽기 경로를 붙인 상태다.
- `APP_DATA`는 없어야 하는 쓰레기 데이터가 아니라, 현재는 데모 fallback 역할도 한다.
- Supabase를 붙였다고 해서 모든 기능이 자동으로 실서비스가 되는 것은 아니다. 읽기, 쓰기, 사용자 상태, 정책을 화면마다 따로 연결해야 한다.
- 그래서 다음 단계는 새 기능을 추가하는 것보다 `이미 있는 화면을 실제 DB 쓰기 기능으로 바꾸는 작업`이 중심이다.

## 12. 결론
현재 SANTA는 방향성 면에서 올바른 궤도에 올라와 있다. 홈 구조와 DB 1단계 설계는 제품 전략과 맞고, 산/커뮤니티 중심 서비스로 가기 위한 기반도 이미 깔렸다.

지금 필요한 것은 대규모 재설계가 아니라, 현재 구조를 기준으로 `게시글 쓰기`, `좋아요`, `저장`, `프로필`, `배지`를 차례로 실제 데이터와 연결하는 일이다. 이 순서를 지키면 동행 기능에 끌려가지 않고, 커뮤니티 중심 제품으로 안정적으로 전환할 수 있다.
